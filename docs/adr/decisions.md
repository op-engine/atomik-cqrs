# Architecture Decision Records

Decisions made in atomik-cqrs, why they were made, and what alternatives were considered.

---

## ADR-01: CQRS + Event Sourcing as the core model

**Status:** Accepted

**Context:** Financial systems need a complete audit trail, safe concurrent writes to the same aggregate, and the ability to reconstruct past state for debugging or compliance. A traditional CRUD model discards historical state and requires pessimistic locking for concurrent writers.

**Decision:** The library models writes as immutable domain events appended to an ordered stream, and reads as replays of that stream. Commands produce events; queries read projections or replay history.

**Consequences:** Single-aggregate transactions only (cross-aggregate operations require sagas). Read models must be built by the consuming application. These are intentional constraints that keep the runtime simple and let consumers choose their consistency model.

---

## ADR-02: Vtable-based adapter interface over comptime generics

**Status:** Accepted

**Context:** The library needs to support multiple storage backends (PostgreSQL, SQLite, in-memory) and allow consumers to swap them at runtime without recompiling the consuming application.

**Decision:** `EventStoreAdapter` is a vtable struct (`*anyopaque` context pointer + function pointers). Concrete adapters expose a `to_adapter()` method that populates the vtable.

**Alternatives considered:** Comptime generics (`fn doSomething(adapter: anytype)`). Rejected because the concrete type leaks into every call site; a consumer cannot store an adapter in a struct field without parameterizing that struct. The vtable has no type parameter; it can be stored, passed, and swapped at runtime.

---

## ADR-03: Parameterized SQL for all database access

**Status:** Accepted

**Context:** Financial systems are high-value targets for SQL injection. The library builds SQL statements that include tenant IDs, aggregate IDs, and user-controlled event data.

**Decision:** All database calls go through `PQexecParams` with the SQL template and parameter values passed separately. String values are never interpolated into SQL text. Dynamic query conditions (e.g., `query_impl` filters) append `$N` placeholders and build a parallel args array.

**Consequences:** The library depends on libpq's parameterized query support. Adapters for databases that do not support native parameterization (rare) must implement their own escaping layer.

## ADR-04: CSPRNG-backed UUID v4 for all identifiers

**Status:** Accepted

**Context:** Event IDs, aggregate IDs, tenant IDs, and user IDs are all UUIDs generated by the library. Predictable IDs would allow an attacker to guess or enumerate IDs belonging to other tenants.

**Decision:** `generate_uuid()` in `cqrs.zig` always uses a cryptographically secure random source: `getrandom(2)` on Linux, `arc4random_buf` on macOS/BSDs, and `crypto.getRandomValues` (via WASM import) on `wasm32-freestanding`. Version (4) and variant bits are set per RFC 4122.

**Consequences:** UUID generation cannot fail silently; on Linux, `getrandom` is asserted to succeed. Any environment that cannot provide OS entropy is not a supported target.

## ADR-05: Optimistic concurrency via a UNIQUE index on (tenant_id, aggregate_id, version)

**Status:** Accepted

**Context:** Two concurrent command handlers for the same aggregate may both read version N, produce events at version N+1, and attempt to write. Without enforcement, both writes succeed and the event stream has two events at version N+1, which silently corrupts the aggregate.

**Decision:** The schema creates `UNIQUE INDEX idx_events_aggregate ON events(tenant_id, aggregate_id, version ASC)`. A concurrent write at the same version fails with PostgreSQL error 23505. The adapter translates this to `error.OptimisticConcurrencyConflict`, which the caller must handle by reloading state and retrying.

**Consequences:** The consuming application is responsible for retry logic. `InMemoryStore` enforces the same uniqueness constraint in memory (pre-write scan), so unit tests can assert OCC behavior without a real database. The behaviour between the two adapters is therefore consistent.

**Migration note:** Existing deployments running before this ADR was adopted need `CREATE UNIQUE INDEX CONCURRENTLY idx_events_aggregate ON events(tenant_id, aggregate_id, version ASC);` applied manually. Non-concurrent creation locks the table.

## ADR-06: Single-threaded connection pool (known limitation before v1.0)

**Status:** Accepted (with limitations)

**Context:** The `ConnectionPool` uses a spinlock to protect connection slot initialization. Once all slots are allocated, all callers share `connections[0]` without synchronization. Two concurrent callers would issue queries on the same `Connection` simultaneously, causing interleaved libpq calls.

**Decision:** The current pool is safe for single-threaded callers only; the primary deployment target is `wasm32-freestanding` Cloudflare Workers, which handle one request at a time.

**Before supporting multi-threaded callers:** Replace the spinlock with a semaphore-guarded queue. `get_connection` should block until a `Connection` is truly idle; `release_connection` should signal the semaphore. The `active_count / connections[0]` fallback must be removed.

## ADR-07: Bridge pattern for libpq (mock vs. real)

**Status:** Accepted

**Context:** CI environments may not have `libpq-dev` installed, and tests must not require a running PostgreSQL instance.

**Decision:** `postgres_pool.zig` imports its libpq backend via a named Zig module called `libpq`. The build system wires either `libpq_mock.zig` (unit tests) or `libpq.zig` (integration tests) into that module name at compile time — no source file needs to change. `libpq_mock.zig` returns `CONNECTION_BAD` and `PGRES_FATAL_ERROR` for every call. Unit tests exercise error propagation without a real database; integration tests (in `src/postgres_integration_test.zig`) verify correct SQL behaviour against a live Postgres 17 instance via the real C library.

**Consequences:** On macOS, libpq is keg-only (Homebrew does not symlink it to the default search path). The Makefile detects the prefix via `brew --prefix libpq` and passes `-Dlibpq-prefix` to `zig build` automatically. Linux CI installs `libpq-dev` via apt-get.

## ADR-08: Fixed-buffer allocator for WASM edge workers

**Status:** Accepted

**Context:** The `wasm32-freestanding` target has no OS allocator (`malloc`, `mmap`, heap growth). All memory must come from a pre-allocated region.

**Decision:** `edge/worker_main.zig` allocates a 256 KB static buffer and serves all per-request allocations from a `FixedBufferAllocator`. After writing the response to `output_buf` (a separate static buffer), `fba.reset()` reclaims all per-request memory in O(1). This makes the allocator an arena: fast bump allocation, free everything at once at request end.

**Consequences:** A single request that allocates more than 256 KB will trap. This is acceptable for the demo harness; production workloads with large event payloads should tune the heap size or use a native target.

## ADR-09: Snapshot store as a separate vtable

**Status:** Accepted

**Context:** Aggregates with long histories incur O(N) replay cost on every command. For aggregates expected to accumulate thousands of events (e.g., a long-lived financial account), this becomes a performance cliff. Snapshots address this by persisting aggregate state at a known version so only incremental events need to be replayed.

**Decision:** `SnapshotStore` is a separate vtable (`*anyopaque` + `load_fn` / `save_fn`) rather than extending `EventStoreAdapter`. Snapshots are stored in a dedicated `snapshots` table with a `UNIQUE (tenant_id, aggregate_id)` constraint — only the most recent snapshot is kept. `EventStoreAdapter.get_events` accepts an `after_version: u32` parameter (`0` = full history); the caller passes `snapshot.version` to skip already-captured events.

The load path follows a two-step pattern the application owns:
```
snap = snapshot_store.load(tenant, agg_id, agg_type)        // may be null
events = adapter.get_events(tenant, agg_id, agg_type, snap?.version ?? 0)
agg.load_from_history_with_snapshot(snap, events, apply_snap, apply_event)
```

**Alternatives considered:** Folding `load_snapshot`/`save_snapshot` into `EventStoreAdapter`. Rejected because not all backends need snapshots, and keeping the vtables separate allows each to be swapped independently.

**Consequences:** The application must decide when to take a snapshot (e.g., every N events). The library provides the mechanism; snapshot frequency is policy. `InMemorySnapshotStore` is provided for tests — it stores at most one snapshot per aggregate (same semantics as the Postgres adapter).

## ADR-10: Migration tool reads .sql files from ./migrations in filename order

**Status:** Accepted

**Context:** The library needs a way to evolve its schema across deployments without requiring the consuming application to manage DDL manually.

**Decision:** `atomik-migrate` (built via `zig build migrate`) scans `./migrations/*.sql` in lexicographic filename order, applies any file not yet recorded in `schema_migrations`, and records the filename there upon success. Each file is split on `;` and each statement is executed individually, so multi-statement files work correctly. The tool is idempotent: re-running it skips already-applied files.

Filename convention: `NNN-description.sql` (e.g., `001-initial-schema.sql`, `002-add-snapshots.sql`). Lexicographic order on the numeric prefix guarantees consistent application order across environments.

**Consequences:** The migration tool runs against `ATOMIK_DATABASE_URL`. Statement-level atomicity (not file-level) means a partially-applied file is possible if a statement fails mid-file; re-running will attempt the whole file again. For production use, each migration file should be a single atomic DDL statement or wrapped in an explicit `BEGIN`/`COMMIT`.

## ADR-11: TypeScript/Hyperdrive persistence adapter for Cloudflare Workers, bypassing native EventStoreAdapter

**Status:** Accepted (fork-specific — mnhpub/atomik-cqrs)

**Context:** The consuming application (Betty; see `docs/decisions/ADR-003-atomik-cqrs-event-sourcing-backend.md` in that repo) deploys the edge WASM harness (`edge/`, see ADR-08) as an actual production event-store, not just a demo. But `EventStoreAdapter`'s only real backend, `postgres_pool.zig`, requires libpq (ADR-07), which needs native OS sockets — `wasm32-freestanding` has none, and Cloudflare's `connect()` socket API (what Hyperdrive-backed drivers use) is only reachable from JavaScript, not from code running inside the WASM sandbox. `edge/worker_main.zig`'s own comment already says as much: "no real DB connection is possible from a demo running purely in the Worker's WASM sandbox."

**Decision:** For this deployment, persistence is handled entirely on the JavaScript side of the FFI boundary — the same `worker.js` shim already used for `fill_random_bytes` (ADR-04) — via a Hyperdrive-pooled Postgres connection (`postgres.js`). The WASM module is used only for what doesn't require I/O: command validation, event application (`Aggregate.load_from_history`), UUID generation. `EventStoreAdapter`'s libpq-based Postgres adapter is not used in this deployment at all.

**Consequences:**
- The event schema (`migrations/*.sql`) stays shared with upstream — only the code path that reads/writes it diverges.
- ADR-05's OCC guarantee (`UNIQUE(tenant_id, aggregate_id, version)`, PG error `23505` → conflict) must be reimplemented in TypeScript, catching the unique-violation and translating it to mirror `postgres_pool.zig`'s behavior exactly, so retry semantics don't diverge from the native adapter's contract.
- ADR-03's parameterized-query requirement carries over in spirit: the TS layer must use `postgres.js`'s parameterized query form, never string-built SQL, for the same SQL-injection reasons.
- ADR-09's `SnapshotStore` is not reused; if snapshotting is needed here, it must be reimplemented in TS against the same `snapshots` table shape, or deferred until replay cost becomes a real problem.
- This decision is fork-specific and does not apply to native (non-Workers) deployments, which should keep using the existing libpq-backed `EventStoreAdapter` as-is.

**Upstream note:** If a general-purpose WASM-to-Postgres bridge is ever built (a socket-proxy shim letting Zig code drive `ConnectionPool` from inside `wasm32-freestanding` itself), it should go upstream as a PR against `opengineorg/atomik-cqrs` rather than stay fork-only — it would remove the need for this split entirely. Track divergence against upstream so this ADR doesn't silently rot once that lands.
