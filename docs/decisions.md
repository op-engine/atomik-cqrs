# Architecture Decision Records

Decisions made in atomik-cqrs, why they were made, and what alternatives were considered.

---

## ADR-01 — CQRS + Event Sourcing as the core model

**Status:** Accepted

**Context:** Financial systems need a complete audit trail, safe concurrent writes to the same aggregate, and the ability to reconstruct past state for debugging or compliance. A traditional CRUD model discards historical state and requires pessimistic locking for concurrent writers.

**Decision:** The library models writes as immutable domain events appended to an ordered stream, and reads as replays of that stream. Commands produce events; queries read projections or replay history.

**Consequences:** Single-aggregate transactions only (cross-aggregate operations require sagas). Read models must be built by the consuming application. These are intentional constraints that keep the runtime simple and let consumers choose their consistency model.

## ADR-02 — Vtable-based adapter interface over comptime generics

**Status:** Accepted

**Context:** The library needs to support multiple storage backends (PostgreSQL, SQLite, in-memory) and allow consumers to swap them at runtime without recompiling the consuming application.

**Decision:** `EventStoreAdapter` is a vtable struct (`*anyopaque` context pointer + function pointers). Concrete adapters expose a `to_adapter()` method that populates the vtable.

**Alternatives considered:** Comptime generics (`fn doSomething(adapter: anytype)`). Rejected because the concrete type leaks into every call site — a consumer cannot store an adapter in a struct field without parameterizing that struct. The vtable has no type parameter and can be stored, passed, and swapped at runtime.

## ADR-03 — Parameterized SQL for all database access

**Status:** Accepted

**Context:** Financial systems are high-value targets for SQL injection. The library builds SQL statements that include tenant IDs, aggregate IDs, and user-controlled event data.

**Decision:** All database calls go through `PQexecParams` with the SQL template and parameter values passed separately. String values are never interpolated into SQL text. Dynamic query conditions (e.g., `query_impl` filters) append `$N` placeholders and build a parallel args array.

**Consequences:** The library depends on libpq's parameterized query support. Adapters for databases that do not support native parameterization (rare) must implement their own escaping layer.

## ADR-04 — CSPRNG-backed UUID v4 for all identifiers

**Status:** Accepted

**Context:** Event IDs, aggregate IDs, tenant IDs, and user IDs are all UUIDs generated by the library. Predictable IDs would allow an attacker to guess or enumerate IDs belonging to other tenants.

**Decision:** `generate_uuid()` in `cqrs.zig` always uses a cryptographically secure random source: `getrandom(2)` on Linux, `arc4random_buf` on macOS/BSDs, and `crypto.getRandomValues` (via WASM import) on `wasm32-freestanding`. Version (4) and variant bits are set per RFC 4122.

**Consequences:** UUID generation cannot fail silently — on Linux, `getrandom` is asserted to succeed. Any environment that cannot provide OS entropy is not a supported target.

## ADR-05 — Optimistic concurrency via a UNIQUE index on (tenant_id, aggregate_id, version)

**Status:** Accepted

**Context:** Two concurrent command handlers for the same aggregate may both read version N, produce events at version N+1, and attempt to write. Without enforcement, both writes succeed and the event stream has two events at version N+1, which silently corrupts the aggregate.

**Decision:** The schema creates `UNIQUE INDEX idx_events_aggregate ON events(tenant_id, aggregate_id, version ASC)`. A concurrent write at the same version fails with PostgreSQL error 23505. The adapter translates this to `error.OptimisticConcurrencyConflict`, which the caller must handle by reloading state and retrying.

**Consequences:** The consuming application is responsible for retry logic. The `InMemoryStore` test double does not yet enforce version uniqueness — tests that assert OCC behavior must use a real PostgreSQL adapter.

**Migration note:** Existing deployments running before this ADR was adopted need `CREATE UNIQUE INDEX CONCURRENTLY idx_events_aggregate ON events(tenant_id, aggregate_id, version ASC);` applied manually. Non-concurrent creation locks the table.

## ADR-06 — Single-threaded connection pool (known limitation before v1.0)

**Status:** Accepted (with limitations)

**Context:** The `ConnectionPool` uses a spinlock to protect connection slot initialization. Once all slots are allocated, all callers share `connections[0]` without synchronization. Two concurrent callers would issue queries on the same `Connection` simultaneously, causing interleaved libpq calls.

**Decision:** The current pool is safe for single-threaded callers only — the primary deployment target being `wasm32-freestanding` Cloudflare Workers, which handle one request at a time.

**Before supporting multi-threaded callers:** Replace the spinlock with a semaphore-guarded queue. `get_connection` should block until a `Connection` is truly idle; `release_connection` should signal the semaphore. The `active_count / connections[0]` fallback must be removed.

## ADR-07 — Bridge pattern for libpq (mock vs. real)

**Status:** Accepted

**Context:** CI environments may not have `libpq-dev` installed, and tests must not require a running PostgreSQL instance.

**Decision:** `postgres_pool.zig` imports its libpq backend via a compile-time constant (`libpq_mock.zig` by default). The mock returns `CONNECTION_BAD` and `PGRES_FATAL_ERROR` for every call. Tests exercise error propagation through the pool and adapter layers without a real database. Switching to the real backend requires changing one import and adding a linker flag.

**Consequences:** Tests verify error propagation, not correct SQL behavior. SQL correctness is validated against a real PostgreSQL instance in integration tests (not yet automated — tracked as a roadmap item).

## ADR-08 — Fixed-buffer allocator for WASM edge workers

**Status:** Accepted

**Context:** The `wasm32-freestanding` target has no OS allocator (`malloc`, `mmap`, heap growth). All memory must come from a pre-allocated region.

**Decision:** `edge/worker_main.zig` allocates a 256 KB static buffer and serves all per-request allocations from a `FixedBufferAllocator`. After writing the response to `output_buf` (a separate static buffer), `fba.reset()` reclaims all per-request memory in O(1). This makes the allocator an arena: fast bump allocation, free everything at once at request end.

**Consequences:** A single request that allocates more than 256 KB will trap. This is acceptable for the demo harness; production workloads with large event payloads should tune the heap size or use a native target.
